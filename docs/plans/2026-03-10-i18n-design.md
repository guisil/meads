# i18n Design — MEADS Project

**Date:** 2026-03-10 (original), 2026-03-16 (revised)
**Status:** Design revised, ready for implementation
**Target languages:** English (default), Portuguese (pt) — later: Spanish (es), Italian (it), Polish (pl)

---

## Scope

**Entrant-facing views and emails only.** Admin views, login/set-password views, and
judge views stay English-only.

### Views to translate

| View | Route | Notes |
|------|-------|-------|
| `MainLayout` | (shared) | Nav items, language toggle, logout |
| `ProfileView` | `/profile` | Name, meadery name, country, preferred language |
| `EntrantOverviewView` | `/my-entries` | Cross-competition entry hub |
| `MyEntriesView` | `/competitions/:c/divisions/:d/my-entries` | Main entry management (biggest view) |

### What gets translated

- **UI labels, buttons, column headers, placeholders, tooltips** in the views above
- **Notification messages** (success, error) in the views above
- **Enum display names** (entrant-facing): EntryStatus, Sweetness, Strength, Carbonation
- **MJP category names/descriptions** — via resource bundles keyed by category `code`
- **Category guidance hints** in entry dialog (17 subcategory descriptions)
- **Entrant-facing emails** — magic link, credentials reminder, password reset, submission
  confirmation, credit notification
- **PDF label instructions** — instruction header only (label body stays English)
- **Entrant-facing service error messages** — via `BusinessRuleException` with message keys

### What does NOT get translated

- **LoginView, SetPasswordView** — used by admins only; entrants use magic links
- **Admin-only views** — UserListView, CompetitionListView, CompetitionDetailView,
  DivisionDetailView, DivisionEntryAdminView, MyCompetitionsView
- **Admin-only emails** — password setup, order review alert
- **Custom division categories** — admin-authored free text
- **PDF label body** — international competition labels, category codes are universal
- **Judge-facing views** — deferred to judging module design

## Architecture

### Locale resolution

| Context | Resolution | Fallback |
|---------|-----------|----------|
| **Authenticated views** | `User.preferredLanguage` | Derive from country → English |
| **Entrant emails** | Recipient's `preferredLanguage` | Derive from country → English |
| **PDF label instructions** | Generating user's `preferredLanguage` | English |

### User.preferredLanguage

New nullable field on `User` entity. Migration V16 adds `VARCHAR(5)` column.

- `null` = derive from country using mapping table (see below)
- Explicit value = user's chosen language
- Exposed in `ProfileView` as a dropdown
- On authentication, `VaadinSession` locale set from this field (or derived)

### Country → language mapping (conservative)

| Language | Countries |
|----------|-----------|
| `pt` | PT, BR |
| `es` | ES, MX, AR, CL, CO, PE, VE, EC, UY, PY, BO, CU, DO, HN, SV, GT, NI, CR, PA |
| `it` | IT |
| `pl` | PL |
| `en` | Everything else (default) |

Countries with multiple official languages (CH, BE, etc.) default to English.
Users can always override in their profile.

### I18NProvider implementation

- Implement Vaadin's `I18NProvider` interface as a Spring `@Component`
- Backed by Spring's `MessageSource` (reads `.properties` bundles)
- `getProvidedLocales()` returns supported locales
- Views use `getTranslation("key")` or `getTranslation("key", param1, param2)`
  (inherited from Vaadin `Component`)

### Enum display names

Enums keep their `getDisplayName()` for backward compatibility in admin views. Entrant-facing
views switch from `enum.getDisplayName()` to `getTranslation("entry.status." + enum.name())`.

### Category translations

MJP catalog categories (M1, M1A, M2A, etc.) translated via resource bundles using category
`code` as the key. The `Category` entity and DB table stay unchanged. Views look up
`getTranslation("category." + category.getCode() + ".name")` with fallback to the DB-stored
English name.

Custom division categories (admin-authored via `DivisionCategory`) are NOT translated — they
display as entered by the admin.

### Language toggle

**MainLayout:** Language dropdown or flag icon in the header. Changes the user's
`preferredLanguage` via `UserService`, updates the `VaadinSession` locale, and reloads
the page.

### Vaadin built-in component translations

Vaadin 25 ships with translations for built-in components (DatePicker, Upload, Grid
pagination, etc.) for many locales. The `I18NProvider` locale setting automatically
activates these. No extra work needed.

## BusinessRuleException pattern

New exception class in `app.meads` root package (shared by all modules). **All services**
use `BusinessRuleException` instead of `IllegalArgumentException` / `IllegalStateException`
for business rule violations. This ensures that when admin views are eventually translated,
no service-layer rework is needed.

```java
public class BusinessRuleException extends RuntimeException {
    private final String messageKey;
    private final Object[] params;

    public BusinessRuleException(String messageKey, Object... params) {
        super(messageKey); // message key as fallback
        this.messageKey = messageKey;
        this.params = params;
    }

    public String getMessageKey() { return messageKey; }
    public Object[] getParams() { return params; }
}
```

Services throw:
```java
throw new BusinessRuleException("error.entry.limit-subcategory", division.getMaxEntriesPerSubcategory());
```

Entrant-facing views catch and translate:
```java
catch (BusinessRuleException ex) {
    var message = getTranslation(ex.getMessageKey(), ex.getParams());
    Notification.show(message, 3000, Notification.Position.BOTTOM_START)
        .addThemeVariants(NotificationVariant.LUMO_ERROR);
}
```

Admin views (for now) use the English bundle value directly — same catch pattern, but
since admin views are not yet translated, they just get the English default.

## Email i18n

`SmtpEmailService` receives the recipient's `preferredLanguage` (or derived locale) and
uses Spring's `MessageSource` to resolve email strings.

### Entrant-facing emails (5 methods)

| Method | Subject key | Recipients |
|--------|------------|------------|
| `sendMagicLink` | `email.magic-link.subject` | Any user |
| `sendCredentialsReminder` | `email.credentials-reminder.subject` | Password users |
| `sendPasswordReset` | `email.password-reset.subject` | Any user |
| `sendSubmissionConfirmation` | `email.submission.subject` | Entrants |
| `sendCreditNotification` | `email.credit.subject` | Entrants |

### Admin-only emails (2 methods — NOT translated)

| Method | Recipients |
|--------|------------|
| `sendPasswordSetup` | Competition admins |
| `sendOrderReviewAlert` | Competition admins |

### Implementation

Email methods receive a `Locale` parameter. The `SmtpEmailService` resolves strings via
`messageSource.getMessage(key, params, locale)` instead of hardcoded strings. The Thymeleaf
template remains the same — it receives the resolved strings as context variables.

---

## PDF label i18n

Only the instruction header at the top of the label page is translated. The label body
(entry details, field names, category codes, disclaimer) stays in English.

`LabelPdfService` receives a `Locale` parameter and resolves instruction strings via
`MessageSource`.

## Decisions log

| Decision | Rationale |
|----------|-----------|
| Entrant-facing views only | Admins are English-comfortable; judges deferred |
| LoginView / SetPasswordView excluded | Entrants use magic links, never see these screens |
| `User.preferredLanguage` in DB | Persists across devices; supports email + PDF locale |
| Country → language derivation (conservative) | Ambiguous countries (CH, BE) default to English |
| `BusinessRuleException` everywhere | Services stay locale-unaware; all views can translate when needed; no rework when expanding i18n to admin views |
| Resource bundles for categories | Avoids DB complexity; MJP catalog is finite and stable |
| Custom categories not translated | Admin-authored free text; deferred until real need |
| Enum translations at view layer | Enums keep `getDisplayName()` for admin views |
| PDF label body stays English | International competition; category codes are universal |
| Single bundle file per locale | Simpler than separate category/message bundles |
| Start with EN + PT, design for 5 | Country mapping and bundle structure support expansion |
