# i18n Design — MEADS Project

**Date:** 2026-03-10
**Status:** Design complete, implementation deferred
**Target languages:** English (default), Portuguese (pt)

---

## Scope

**Entrant-facing views only.** Admin and judge views stay English-only for now.

### Views to translate

| View | Route | Notes |
|------|-------|-------|
| `LoginView` | `/login` | Email, magic link, credentials |
| `MainLayout` | (shared) | Nav items: My Entries, My Profile, Logout |
| `ProfileView` | `/profile` | Name, meadery name, country fields |
| `EntrantOverviewView` | `/my-entries` | Cross-competition entry hub |
| `MyEntriesView` | `/competitions/:c/divisions/:d/my-entries` | Main entry management (biggest view) |
| `SetPasswordView` | `/set-password` | Password setup via token |

### What gets translated

- **UI labels, buttons, column headers, placeholders** in the views above
- **Notification messages** (success, error) in the views above
- **Enum display names** (entrant-facing): EntryStatus, DivisionStatus, Sweetness, Strength, Carbonation
- **MJP category names/descriptions** — via resource bundles keyed by category `code` (M1, M1A, M2A, etc.)

### What does NOT get translated (for now)

- Admin-only views: UserListView, CompetitionListView, CompetitionDetailView, DivisionDetailView, DivisionEntryAdminView
- Custom division categories (admin-authored free text) — deferred until it becomes a real need
- Service-layer error messages (IllegalArgumentException, ConstraintViolationException) — admin-facing or technical
- Judge-facing views — judging module not yet built; i18n will be considered during its design

## Architecture: Vaadin I18NProvider + Spring MessageSource

### Locale resolution

1. **Default:** Browser locale auto-detected by Vaadin
2. **Override:** UI language switcher (in MainLayout header area), stores preference in cookie or localStorage
3. **No DB change** — no language column on User entity

Browser locale is based on user's browser/OS language settings (not geolocation). A Portuguese
user with a Portuguese browser gets Portuguese automatically. A Portuguese user with an English
browser can switch via the UI.

### Resource bundles

```
src/main/resources/
  messages.properties          # English (default)
  messages_pt.properties       # Portuguese
  categories.properties        # MJP category names/descriptions (English)
  categories_pt.properties     # MJP category names/descriptions (Portuguese)
```

Key naming convention:
```properties
# messages.properties
login.email.label=Email
login.magic-link.button=Get Login Link
nav.my-entries=My Entries
entry.status.DRAFT=Draft
entry.sweetness.DRY=Dry
division.status.REGISTRATION_OPEN=Registration Open
category.M1.name=Traditional Mead
```

### I18NProvider implementation

- Implement Vaadin's `I18NProvider` interface as a Spring `@Component`
- Backed by Spring's `MessageSource` (which reads the `.properties` bundles)
- `getProvidedLocales()` returns `[Locale.ENGLISH, new Locale("pt")]`
- Views use `getTranslation("key")` (inherited from Vaadin `Component`)

### Enum display names

Enums keep their `getDisplayName()` for backward compatibility in admin views. Entrant-facing
views switch from `enum.getDisplayName()` to `getTranslation("entry.status." + enum.name())`.

### Category translations

MJP catalog categories (M1, M1A, M2A, etc.) translated via resource bundles using category
`code` as the key. The `Category` entity and DB table stay unchanged. Views look up
`getTranslation("category." + category.getCode() + ".name")` with fallback to the DB-stored
English name.

Custom division categories (admin-authored via `DivisionCategory`) are NOT translated — they
display as entered by the admin. This is acceptable because custom categories are rare and
admin-defined.

### Language switcher

- Placed in `MainLayout` (visible to all authenticated users)
- Stores selected locale in a cookie or `VaadinSession`
- On change, reloads the current page with the new locale
- Shows only for entrant-facing contexts, or always visible (simple flag/language icon)

### Vaadin built-in component translations

Vaadin 25 ships with translations for built-in components (DatePicker, Upload, Grid pagination,
etc.) for many locales including Portuguese. The `I18NProvider` locale setting automatically
activates these. No extra work needed.

## Implementation notes (for when this is picked up)

### Estimated string count

~100-120 strings across the 6 entrant-facing views + enums + categories.

### Migration order (suggested)

1. Wire up I18NProvider + MessageSource infrastructure (no visible change)
2. Extract English strings from entrant-facing views into `messages.properties`
3. Add Portuguese translations in `messages_pt.properties`
4. Add language switcher to MainLayout
5. Add category translations (`categories.properties` / `categories_pt.properties`)
6. Test with Karibu (locale switching, fallback behavior)

### Testing considerations

- Karibu tests can set locale via `UI.getCurrent().setLocale()`
- Test that missing translation keys fall back to English (default bundle)
- Test language switcher stores and restores preference
- Test enum translations render correctly in grids and forms

### Dependencies on other work

- If deployment happens first, resource bundles deploy as part of the JAR (no extra config)
- If email sending is implemented first, email templates may also need i18n (separate concern)
- Judging module will need its own i18n pass when designed

## Decisions log

| Decision | Rationale |
|----------|-----------|
| Entrant-facing only | Admins are English-comfortable; judges deferred to judging module |
| Browser locale + UI switcher | No DB change needed; covers auto-detection + manual override |
| Cookie/localStorage for override | Simpler than User entity change; persists across sessions |
| Resource bundles for categories | Avoids DB complexity (translation tables); MJP catalog is finite and stable |
| Custom categories not translated | Admin-authored free text; deferred until real need arises |
| Enum translations at view layer | Enums keep `getDisplayName()` for admin views; entrant views use bundle lookup |
| Vaadin I18NProvider + Spring MessageSource | Standard approach; integrates with Vaadin built-in component translations |
